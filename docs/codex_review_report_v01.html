암호화 설계는 올바르다 — AES-256-GCM + HMAC-SHA256 해시 구조는 적절하다. 문제는 불완전한 구현이다: 스키마에 컬럼이 없고, 평문을 암호문 옆에 동시 저장하고, 업로드 경로를 빠뜨렸다. 또한 테스트 자격증명이 git에 커밋되어 있어 즉각 로테이션이 필요하다. 설계를 폐기할 필요는 없다 — 구현 5단계를 완성하면 목표한 보안 수준에 도달한다.

P1 Critical Findings

5건 — 즉시 수정
P1-01 로그인 자격증명 git에 커밋됨 .env.test:5-6 즉각 조치

email과 password가 .env.test에 평문으로 커밋되어 있다. 저장소 읽기 권한이 있는 누구든 해당 계정으로 로그인 가능하다. E2E spec 파일의 fallback 상수(TEST_EMAIL, TEST_PASSWORD)도 동일 문제다.

조치: .env.test 파일에서 자격증명 제거 → git-secrets 또는 BFG로 git history 정리 → 관련 계정 비밀번호 즉시 로테이션.

⚠ 지금 이 순간도 노출 중 — 즉시 로테이션 필요
P1-02 암호화 컬럼 마이그레이션 스키마에 없음 schema.sql · scripts/migrate.ts

API 코드는 member_name_enc, member_name_hash, submitter_enc, name_enc, name_hash에 INSERT하지만, 이 컬럼들이 schema.sql과 scripts/migrate.ts에 존재하지 않는다. 클린 체크아웃 후 마이그레이션 실행 시 모든 INSERT가 column does not exist로 실패한다.

조치: schema.sql에 5개 컬럼 ALTER TABLE 추가, scripts/migrate.ts에 실행 가능한 DDL 포함.

P1-03 평문 PII가 암호문 옆에 동시 저장됨 offerings/route.ts:45-46 · feedback/route.ts · members/route.ts

member_name, submitter, name 평문 컬럼이 암호화 컬럼과 함께 INSERT된다. DB가 노출되면 암호화된 값이 있더라도 원본 평문이 그대로 읽힌다 — 암호화의 의미가 없다.

조치: 3개 route의 INSERT에서 평문 컬럼 제거. 조회 시 decrypt()로 복원. 기존 데이터는 마이그레이션 스크립트로 암호화 후 평문 컬럼 DROP.

P1-04 Excel 업로드 경로에 암호화 미적용 app/api/upload/route.ts

app/api/upload/route.ts는 Excel에서 추출한 member_name을 평문 그대로 INSERT한다. 업로드로 입력된 헌금 데이터는 암호화·해시가 전혀 없는 상태로 DB에 저장된다.

조치: 업로드 route에 encrypt() + hashForSearch() 적용. offerings API와 동일한 패턴 사용.

P1-05 Playwright CI 미구성 — 클린 체크아웃 빌드 실패 package.json · playwright.config.ts · .github/workflows/e2e.yml

@playwright/test가 package.json devDependencies에 없다. npm run test:e2e 스크립트도 미정의다. CI 워크플로우가 존재하지만 실행 시 즉시 실패한다.

조치: npm install -D @playwright/test 실행 → package.json의 scripts에 "test:e2e": "playwright test" 추가.

P2 Advisory Findings

5건 — 단·중기 수정
P2-01 CI E2E 서버에 ENCRYPTION_KEY 미주입 .github/workflows/e2e.yml:43-45

CI에서 기동하는 Next.js 서버가 ENCRYPTION_KEY를 받지 못한다. 인증된 API 호출 시 encrypt()가 500을 반환 — E2E 테스트 전체가 의미 없어진다.

조치: GitHub Secrets에 ENCRYPTION_KEY 등록 → e2e.yml의 서버 기동 step에 env: ENCRYPTION_KEY: ${{ secrets.ENCRYPTION_KEY }} 추가.

P2-02 CSP에 unsafe-inline + unsafe-eval 동시 허용 next.config.mjs:8

Content-Security-Policy에 'unsafe-inline'과 'unsafe-eval' 두 지시어가 함께 있다. 인젝션 취약점이 생기면 이 CSP는 XSS를 막지 못한다. 헤더가 있어도 실질적 방어가 없는 상태.

조치: nonce 또는 hash 기반 CSP로 교체. 개발 환경 전용 완화 규칙은 NODE_ENV === 'development' 조건으로 분리.

P2-03 헌금대장·수입내역·지출내역 200행 하드 리밋 app/account-book/page.tsx:13 · income/page.tsx · expense-view/page.tsx

API 기본 limit=200이 3개 페이지에 적용되어 있다. 꿈꾸는교회 연간 헌금 1,186건+ 기준, 200건 초과분이 합계에서 누락된다. 재정 보고서 숫자가 틀려진다.

조치: 집계용 API 파라미터 limit=10000 또는 pagination 구현. 페이지 레벨 집계가 아닌 DB 집계 함수(SUM) 사용 권장.

P2-04 피드백 미인증 접근 테스트 로직 오류 e2e/security.spec.ts:54-56

app/api/feedback/route.ts POST에 세션 체크가 없다. 미인증 요청이 실제로 201을 반환한다. 테스트는 401을 기대하므로 항상 실패한다.

조치: (A) feedback POST에 getServerSession 인증 추가 — 피드백을 익명으로 받으려면 (B) 테스트 expected를 201로 변경하고 설계 의도 명시.

P2-05 XSS 테스트가 저장·렌더링 없이 종료 e2e/security.spec.ts:89-93

테스트가 입력 필드에 XSS 페이로드를 입력하고 저장 없이 검사를 완료한다. 저장형(Stored) XSS가 있어도 이 테스트는 항상 통과한다.

조치: dialog 이벤트 리스너 등록 → 행 저장(submit) → 목록 페이지 이동 → page.on('dialog')가 실행되지 않음 확인.
